# Render デプロイの仕組み（初学者向け）

作問支援システムを Render で公開・運用するための仕組みを、体系立ててまとめる。

## 結論（現在の設定は正しい）

| 項目 | 現在の値 | 判定 |
|---|---|---|
| 永続ディスクのマウントパス | `/data`（1GB） | ✅ |
| 環境変数 `DATABASE_PATH` | `/data/sakumon.db` | ✅ |

DBファイルが「消えない領域（`/data`）」の中にあるため、**デプロイしてもデータは残る**。

| 用語 | ひとことで言うと |

| **Render** | アプリを24時間動かしてくれるクラウドの大家さん |

| **永続ディスク（Disk）** | 部屋とは別にある「消えない引き出し」 |

## 1. 全体像 — Render は何をしているのか

`git push` すると、Render はこう動く。

```
あなたのPC                Render のクラウド
──────────              ───────────────────
git push  ──────────▶   ① コードを取ってくる
                        ② Build（pip install で部品を用意）
                        ③ 新しい「箱（部屋）」を作る
                        ④ その中で uvicorn を起動 ← アプリ稼働！
                        ⑤ https://sakumon.onrender.com で公開
```

Render は「コードを、ネット上の部屋で動かしてくれる」存在。

---

## 2. なぜ「データが消える」問題があるのか 🏨

**箱（部屋）の中の普通のファイルは使い捨て**。

> 🏨 ホテルのたとえ
> コンテナ（箱）はホテルの部屋。デプロイのたびに「チェックアウト → 新しい部屋にチェックイン」する。
> 部屋は毎回掃除されて新品に戻るので、前の部屋の机に置いた私物（＝DBファイル）は消える。

対策せずに `data/sakumon.db` に保存すると：

```
1回目のデプロイ → 部屋A で児童のデータが貯まる
2回目のデプロイ → 部屋B（新品）に交代 → 部屋Aごと消滅 → データ全消し
```

これが「Render で SQLite が消える」の正体。コードが悪いのではなく、箱が使い捨てだから。

---

## 3. 永続ディスク = 消えない引き出し 🔐

> 🔐 貸金庫のたとえ
> ディスクは、部屋とは別の場所にある貸金庫。部屋を変えても（デプロイしても）金庫はずっと残る。
> 新しい部屋に移るたびに、Render がこの金庫を部屋の中の `/data` という扉につなぎ直す。

```
     箱（毎回作り直される・使い捨て）
   ┌─────────────────────────────┐
   │  アプリ（uvicorn）            │
   │                             │
   │   /data  ←──────────┐       │   ← この扉の先が…
   └─────────────────────┼───────┘
                         │
              ┌──────────▼───────────┐
              │  永続ディスク 1GB      │   ← 消えない金庫（デプロイしても残る）
              │  sakumon.db          │
              │  sakumon.db-wal      │
              └──────────────────────┘
```

**ポイント：`/data` の中に置いたものだけが生き残る。** それ以外の場所は毎回消える。

- **Mount path `/data`** = 「箱の中のどこに金庫の扉をつけるか」という住所
- DB を `/data/sakumon.db`＝金庫の中に置いているので消えない

---

## 4. 環境変数 = 箱に貼る「設定メモ」 📝

コードに直接書きたくない情報を、箱の外から渡す仕組み。

**① 秘密だから（`ANTHROPIC_API_KEY`）**
APIキーをコードに書いて GitHub に push すると世界中に鍵が漏れる。だから Render の「Environment」に貼り、コードは環境変数から読むだけにする。
→ ローカルでは `.env` ファイルが同じ役割（だから `.env` は `.gitignore` 済み）。

**② 場所によって値を変えたいから（`DATABASE_PATH`）**

| 環境 | DBを置きたい場所 |
|---|---|
| ローカル（自分のPC） | `backend/../data/sakumon.db` |
| Render | `/data/sakumon.db`（金庫の中）|

コードを2種類に分けず、`DATABASE_PATH` という設定メモで切り替える。

---

## 5. このアプリで実際どうつながっているか 🔌

`backend/database.py` の該当コード：

```python
_env_path = os.environ.get("DATABASE_PATH")          # ① 設定メモを読む
DB_PATH = Path(_env_path) if _env_path else Path(__file__).parent.parent / "data" / "sakumon.db"
#          └ メモがあればそれを使う      └ なければローカル用の既定の場所
if not DB_PATH.parent.exists():
    DB_PATH.parent.mkdir(parents=True, exist_ok=True) # ② 保存先フォルダがなければ作る
```

```
ローカル（DATABASE_PATH なし）  → data/sakumon.db に保存
Render（DATABASE_PATH=/data/…）→ /data/sakumon.db（金庫）に保存
```

同じコードなのに、環境変数1つで保存先が変わる。

## 6. デプロイのたびに何が起きるか（ライフサイクル）🔄

```
git push
   │
   ▼
① Build:  pip install -r backend/requirements.txt   （部品を用意）
   │
   ▼
② 新しい箱を作る（前の箱は捨てる＝中のファイルは消える）
   │
   ▼
③ 金庫 /data を新しい箱につなぎ直す  ← ここでデータが引き継がれる
   │
   ▼
④ Start:  uvicorn main:app --host 0.0.0.0 --port $PORT   （アプリ起動）
   │
   ▼
⑤ 公開 → https://sakumon.onrender.com
```

- `$PORT` も環境変数。Render が「このポートで待ち受けてね」と箱に教えてくる番号。
- `--host 0.0.0.0` は「箱の外からの通信を受け付ける」の意味（ないと外から繋がらない）。
- `--reload` はローカル開発専用。本番では付けない。

---

## 7. まとめ（ローカル vs Render 対応表）

| 項目 | ローカル（自分のPC） | Render（本番） |
|---|---|---|
| アプリの起動 | `uvicorn main:app --reload --port 8000` | `uvicorn main:app --host 0.0.0.0 --port $PORT` |
| APIキー | `.env` ファイル | Environment の `ANTHROPIC_API_KEY` |
| DBの場所 | `data/sakumon.db` | `/data/sakumon.db`（永続ディスク内）|
| DBが消えるか | 消えない（自分のPC） | 消えない（金庫に入れてあるから）|

## 運用メモ

- **DBのバックアップ**：管理者画面の CSV エクスポート（`/admin/api/export/csv`）で研究データを定期的に手元へ保存できる。
- **スキーマ変更**：列の追加は `ALTER TABLE ... ADD COLUMN`（非破壊）で行う。`database.py` の `_migrate()` が起動時に実行する。テーブルの作り直し（DROP）はデータが消えるので避ける。
